# Спека: E2E-мониторинг оплаты AIM Website

Черновик 2026-07-07 (Claude + Анка). Статус: на обсуждение с программистом.

## Зачем

Платёжные баги — самая частая тема чата за последние недели: ошибка оплаты без
email (22.06), «у нас не работает форма оплаты» (01.07), зависшая оплата у
пользовательницы (06.07), `invalid_product` в NocoDB (07.07). QA-контур сейчас
проверяет только ссылки/скриншоты — сам платёжный поток не мониторится никем.
Об инцидентах узнают пользователи раньше команды.

## Два уровня проверки

### Уровень 1 — синтетический (без денег, каждые 2–4 часа)

Playwright-сценарий в AIM Site Agent Evaluation:
1. Открыть страницу оплаты (staging и production).
2. Выбрать тариф → заполнить форму валидными тестовыми данными
   (помеченными, чтобы lead-relay/CRM их отфильтровывали: например email
   `qa-monitor@aimindset.org`, ник `@aim_qa_monitor`).
3. Проверить: цены не пустые в обеих валютах (₽/USDT), промокод-поле работает,
   форма отправляется, редирект на платёжный шлюз происходит и URL шлюза
   содержит правильные параметры (order_id, сумма, product code).
4. Стоп перед реальной оплатой.
5. Проверить, что lead-сигнал дошёл (Метрика-цель / lead-relay лог) и помечен
   как тестовый.

### Уровень 2 — реальный платёж (раз в день, минимальная сумма)

Идея Анки: не останавливаться на шлюзе, а реально платить, чтобы проверить
весь круг. Варианты (выбрать с программистом):

- **Вариант А — промокод −100%:** заказ на 0 не всегда проходит через шлюз
  (многие провайдеры отклоняют нулевые суммы) — проверить, поддерживает ли
  наш флоу free-order ветку. Если да — самый дешёвый способ.
- **Вариант Б — промокод −99.9% / технический тариф за ~1 ₽/1 цент:**
  ежедневный реальный микро-платёж картой команды. Плюс: проверяет реальный
  шлюз, вебхуки, статусы. Минус: комиссии, вопрос возвратов, шум в отчётности —
  нужен отдельный `product_code=qa_monitor`, который исключается из выручки.
- **Вариант В — тестовый режим шлюза** (если Stripe/Moneta/OxaPay дают
  sandbox-ключи на staging): реальный флоу без денег, но не ловит проблемы
  production-конфига.

Рекомендация: Б для production (раз в день) + А/В для staging (каждый прогон).

### Что проверяется после оплаты (в обоих уровнях, где применимо)

1. **Redirect:** success-URL открывается, содержит правильный order_id.
2. **Текст:** на success-странице правильное имя/название лаборатории
   (персонализация по имени — то, что Анка просила проверять).
3. **Статус заказа:** в NocoDB появляется запись с прикреплённым Telegram-ником
   (связка ник↔заказ — EPP-212).
4. **Telegram:** уведомление о платеже приходит в правильную часть бота
   (lead-relay → правильные получатели; тестовые платежи должны идти только
   владельцу/в тестовый канал, не всем менеджерам).

### Алерты

- Провал любого шага → Вася шлёт сообщение в группу сайта немедленно
  (не ждёт понедельничного дайджеста), с шагом, скриншотом и URL.
- Успех — тишина (или тихая отметка в reports/latest).

## Что нужно от программиста

1. Решение по варианту уровня 2 (А/Б/В) и создание промокода/тарифа
   `qa_monitor` (связано с EPP-211 — промокод −99% для теста оплат).
2. Фильтрация тестовых лидов/платежей в NocoDB, Метрике и lead-relay
   (по product_code или фиксированному нику).
3. Ревью Playwright-сценария (добавим в aim-site-agent-evaluation,
   `npm run qa:payment`).
