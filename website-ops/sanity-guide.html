1. Назначение документа

Документ объясняет роль Sanity в проекте, описывает структуру данных, показывает, как наполнять базу контентом (через интерфейс и через агентов), и как переиспользовать эти данные на других страницах сайта и вайбкод-лендингах. Цель — единое понимание того, что такое CMS-слой, как с ним работать и как его не сломать.

2. Архитектура: Sanity и сайт — две разные вещи

Система состоит из трёх независимых слоёв:

СлойЧто этоКто меняет
Sanity Content LakeОблачная база данных — набор документов (кейсы, отзывы, спикеры, лаборатории). Размещается на стороне Sanity.API (агент/скрипт) или редактор через Studio
Sanity StudioВеб-редактор поверх базы («админка»): формы, drag&drop блоков, кнопка Publish.Администратор, вручную в браузере
Сайт (Astro)Движок: страницы и компоненты, в которых закодировано, какие данные и в каком виде отображать. Генерирует статичный HTML.Разработчик (код)

Sanity хранит данные; сайт — это движок отображения. Источник истины для контента — Sanity.

Цикл публикации

Правка в Studio→ Publish→ webhook→ пересборка Astro→ обновлённые HTML-страницы

Редактор не программирует — он меняет данные, после чего движок пересобирает страницы автоматически. Участие агента в этот момент не требуется.

3. Зачем база данных, а не markdown в репозитории

Markdown и Sanity — не альтернативы, а разные роли. Markdown удобен для авторской подготовки и переноса (его легко генерировать и хранить в git). Sanity — это рантайм-источник истины, из которого собирается сайт. В проекте уже есть мост между ними — команды import-lab / export-lab.

КритерийMarkdown в репозиторииSanity (CMS)
Кто может правитьТот, кто умеет в git и разметкуЛюбой, через Studio
Попадание на сайтНужен агент, чтобы интерпретировать MD и разместить в кодеPublish → авто-пересборка
Структура и проверкиСвободный текстСхема: обязательные поля, форматы, списки значений
Связи между сущностямиПо соглашению, вручнуюReferences: сущность заведена один раз, подставляется везде
ИзображенияНе переносятся (только пути)Управляемые ассеты (CDN, кроп)
Версионированиеgit historyDraft/Publish + история версий

Вывод

Sanity целесообразно использовать как базу данных и источник истины для кейсов, отзывов, спикеров и лабораторий. Markdown — как формат авторинга и резервного экспорта. Это взаимодополняющие части одного пайплайна, а не конкуренты.

5. Наполнение данными

Есть два способа положить данные в Sanity:

5.1. Вручную через Studio

Администратор открывает Studio, заполняет форму документа, прикрепляет фото, нажимает Publish. Подходит для единичных правок и работы без агента.

5.2. Программно (агент или скрипт)

Запись идёт по HTTP через @sanity/client. Запущенный Studio не требуется. Нужны четыре параметра и токен с правом записи:

ПараметрЗначение
projectIdms4mgrt9
datasetproduction
apiVersion2026-03-29
tokenSANITY_API_TOKEN (права Editor; только локально / в CI, не в браузер)
 
await client.createOrReplace({
  _id: 'case-ai-coaching',   // детерминированный id → повторный запуск не плодит дубли
  _type: 'caseStudy',        // точно как в схеме
  title: 'AI COACHING',      // обязательное, уникальное
  tools: 'Claude · Obsidian',
  filters: ['manager'],       // только из списка схемы
  orderRank: 'a0|',
})

Чем можно нарушить структуру

  • Неверный _type — должен буквально совпадать со схемой; иначе документ не найдётся и не отрисуется.
  • Пропуск обязательных полей — публикация будет отклонена.
  • Дубли идентификатора (title/name) — ломают связь из блока. Защита: детерминированный _id + createOrReplace.
  • Отсутствие _key у элементов массива — Studio выдаёт «Missing keys» и ломает drag&drop.
  • References — это объект {_type:'reference', _ref:'<id>'}, а не строка; ссылка только на существующий документ.
  • Спецформаты: faqItem.title обязан матчить ^[a-z0-9_]+$; описания и юр-страницы — Portable Text, а не строка.
  • createOrReplace перезаписывает документ целиком — для точечной правки используется patch().set().

Образцы рабочего кода записи — в scripts/seed-cases.mjs, seed-reviews.mjs. Чтобы агент делал это корректно и безопасно при произвольном запросе человека, введён скилл /sanity-content (раздел 6).

6. Скилл /sanity-content новое

Скилл-инструкция для агента: наполнять Sanity контентом по запросу человека, понимая намерение и не нарушая структуру.

Что покрывает

УровеньЧто делает
СущностьСоздаёт/обновляет документы: кейсы, отзывы, спикеры, FAQ, цитаты, философию, продукты.
БлокНаполняет блок внутри лаборатории: создаёт нужные сущности и связывает их через references.
ЛабораторияЦелую лабу не дублирует — опирается на существующий MD-пайплайн (import-lab).

Ключевые принципы поведения

  1. Разбор намерения. Понимает свободные формулировки («наполни кейсы, данные возьми здесь»): определяет тип сущности, источник данных, наличие фото. При неоднозначности — уточняет.
  2. Безопасная обработка дублей. Перед записью проверяет существующие документы по идентификатору. Если найден дубль и нет явной команды обновлять — останавливается и спрашивает (обновить / пропустить / создать новый). Молча не перезаписывает.
  3. Валидация до записи. Проверяет обязательные поля, форматы и существование связей. Битый документ не пишет.
  4. План перед записью. Показывает сводку (создать / обновить / пропустить), получает подтверждение, затем пишет.
  5. Фото. Загружает изображения как ассеты (двухшаговая привязка). Отсутствующий файл — повод уточнить, а не пропустить молча.
  6. Целостность. Детерминированный _id, _key для массивов, orderRank для сортируемых типов, patch для частичных правок.

Как вызвать

В сессии агента (в этом репозитории, с настроенным .env): /sanity-content и далее запрос — например, «добавь три кейса, данные в ./my-cases.md, фото в ./photos/». Агент разберёт запрос, покажет план, спросит про дубли и выполнит.

7. Скиллы для агента vs запрограммированные скрипты

Важно различать два типа автоматизации — это не одно и то же.

Скилл (для агента)Запрограммированный скрипт
Что этоИнструкция, которую агент исполняет, понимая контекстДетерминированная программа (npm-команда)
ВходСвободный: текст, файлы, фото, устный запросСтрого заданный формат
Понимает намерениеданет
ГибкостьВысокая (агент адаптируется)Нулевая (делает ровно одно)
Где живёт.claude/commands/*.mdscripts/*.mjs, package.json
Примеры/sanity-content, /sanity-schema, /port-blockimport-lab, export-lab, seed-*.mjs

Скилл может опираться на скрипт (например, /sanity-content для уровня «лаборатория» использует import-lab), но природа у них разная: скрипт не «понимает» человека, скилл — понимает.

Что есть и чего не было

Наполнение целой лабы из MD было закрыто скриптом import-lab. А вот наполнение отдельных сущностей (кейсы/отзывы/спикеры) из произвольного ввода с фото скриптами не покрывалось — только разовые хардкод-seed. Этот пробел и закрывает новый скилл /sanity-content.

8. Где запускать агента: в репозитории или вне

Ответ зависит от направления — запись или чтение.

ОперацияТокенПапка studio/Где запускать
Запись (наполнение контентом)нужен (write)желательнав этом репозитории
Чтение (данные на лендинг)не нуженне нужнагде угодно

Почему запись держим в этом репозитории

Агент пишет безопасно только тогда, когда знает схему: точные _type, обязательные поля, форматы, по какому полю резолвятся ссылки. Эти знания — в studio/schemaTypes/. Во внешнем репозитории их нет; пришлось бы дублировать «контракт схемы» отдельным документом и держать его в синхроне (риск рассинхрона = риск повредить данные). Поэтому запись выполняется в этом репозитории, где есть схемы, рабочие образцы скриптов и токен в .env.

9. Чтение данных на сторонние страницы и вайбкод-лендинги

Чтение проще записи: датасет публичный, токен не нужен. Минимум — клиент и GROQ-запрос.

const sanity = createClient({
  projectId: 'ms4mgrt9', dataset: 'production',
  apiVersion: '2026-03-29', useCdn: true,  // без токена
})
const cases = await sanity.fetch(`
  *[_type == "caseStudy"] | order(orderRank asc){
    title, author, tools, metric, "imageUrl": image.asset->url
  }`)

В проекте это уже работает: вайбкод-лендинги тянут из Sanity навигацию и футер, а тело пока захардкожено. Готовые запросы — в src/lib/queries.ts, их можно переиспользовать. Внешнему сайту, читающему из браузера, дополнительно нужно прописать его origin в CORS-allowlist Sanity.

10. Безопасность: что значит «публичный датасет»

Чтение без токена означает, что любой, кто знает projectId, может прочитать опубликованные документы через API. Это штатная модель headless CMS со статическим сайтом. Важные границы (проверено запросами к API):

Доступно без токенаЗащищено
Опубликованный контент (то же, что и так показано на публичном сайте)Черновики (draft) — без токена не видны
Чтение через API в формате JSONЗапись и удаление — невозможны без write-токена
projectId (он и так зашит в URL картинок сайта)Write-токен — только в .env/CI, в браузер не передаётся

Итог

Публичный датасет = «публичный сайт, только в виде JSON». Новых данных наружу это не открывает: всё доступное и так отрисовано на сайте. В Sanity нет чувствительной информации (персональные данные клиентов, платёжные секреты хранятся в отдельном бэкенде). Если в будущем понадобится хранить непубличный контент — датасет можно перевести в private (тогда токен потребуется на любое чтение, включая сборку сайта и сторонние лендинги).

11. Один контент — разные варианты дизайна

При правильно спроектированной структуре данных один и тот же контент можно отображать в нескольких вариантах дизайна. В проекте данные и отрисовка разделены: сущности (например, кейсы) хранятся в Sanity один раз, а способ их показа определяет компонент сайта.

Поэтому второй (или третий) вариант дизайна блока — это не копия данных, а поле-переключатель на блоке. Движок по значению переключателя выбирает соответствующий компонент-отрисовщик, а данные оба варианта берут одни и те же.

Данные в Sanity (один раз)→ переключатель variant на блоке→ вариант Aвариант B

Требование к структуре: данных в Sanity должно хватать для всех вариантов отображения — схема проектируется универсальной, под все нужные представления сразу.

Реализация: поле variant в схеме блока (через скилл /sanity-schema) → проекция поля в GROQ → выбор компонента по variant на странице. Изменение небольшое именно потому, что данные уже отделены от отрисовки.
