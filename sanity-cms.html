данные, не схема

Наполнение Sanity

Новый /sanity-content нужен для первичного наполнения и точечных правок сущностей. Он работает с документами и references, но не должен менять структуру схем.

ручной вводзапись агентом

Что можно наполнять

  • Standalone-документы: caseStudy, review, speaker, faqItem, quote, philosophyPillar, closingPhrase, product.
  • Блоки лаборатории: создать или найти сущности и связать их в blocks[] через reference-объекты с _key.
  • Целую лабораторию не собирать вручную: для неё остаётся MD-пайплайн import-lab / export-lab.
запись только из repo

Где запускать агента

  • Для записи запускать в ai-mindset-website, где есть studio/schemaTypes/, примеры seed-*.mjs и локальный .env.
  • SANITY_API_TOKEN нужен только для write/import и не должен попадать в браузер, логи, правила или tracked files.
  • Если данные не помещаются в текущую схему, агент останавливается и предлагает вариант, а не меняет файлы схемы.
защита схемы

Схему не трогать

  • Не добавлять, не переименовывать и не удалять поля в studio/schemaTypes/.
  • Не менять validation, типы блоков и структуру blocks[] без отдельной задачи на изменение схемы.
  • Наполнять существующие блоки можно; добавлять, удалять или переставлять типы блоков — только по явной просьбе.
пландубли

План до записи

  • До записи показать сводку: что будет создано, обновлено или пропущено.
  • Дубли искать по id-полю: title или name, затем обновлять реальный _id.
  • Без явной команды на overwrite найденный дубль означает остановку и вопрос человеку.
только чтениелендинги

Читать можно проще

  • Опубликованный dataset читается без токена: это публичный сайт в JSON-форме.
  • Для стороннего лендинга нужен Sanity client, GROQ-запрос и разрешённый origin в Sanity CORS.
  • Черновики, запись и удаление без write-токена недоступны.

основной маршрут

Как создать лабу через Markdown

Самый частый сценарий: агент собирает MD, редактор проверяет в Studio, публикация уходит на staging/new.

01

Скачать свежий общий контент

В Sanity Studio нажать Экспорт общего контента. Полученный common_content.md — единственный источник reference-ID.

02

Собрать lab Markdown

Агент читает create-lab-from-md.md, brief и свежий экспорт. Если нет product code или сущности в Sanity — агент спрашивает, а не угадывает.

03

Импортировать в Studio

Для новой лабы использовать Создать лабу из MD, для существующей — Обновить из MD. После импорта редактор смотрит результат до publish.

04

Проверить staging/new

Publish запускает rebuild. Проверять страницу лабы на staging.aimindset.org/labs/<slug> и затем на new.

Жёсткие правила

Эти правила нельзя восстанавливать из памяти чата, старого export-а или скриншота.

reference-IDSanity

Только свежий common_content.md

  • Reference-блоки заполняются точь-в-точь ID из свежего экспорта.
  • Если сущности нет в экспорте, её нужно создать в Studio и экспортировать файл заново.
  • Пустой reference-блок означает особое import-поведение и используется только осознанно.
оплатаNocoDB

product_code не придумывается

  • Код каждого тарифа приходит из каталога Дана / NocoDB.
  • Код нельзя строить из slug, названия лаборатории или цены в тексте.
  • Sanity хранит код, а финальная цена приходит runtime-проверкой оплаты.
identitydates

Slug и даты задают документ

  • Frontmatter slug задаёт URL и идентичность; имя файла не решает.
  • Даты пишутся строго YYYY-MM-DD.
  • FAQ остаётся плоским списком без категорий.

подключение

API и командная строка

Команды нужны для локальной Studio, clean scratch Studio и repo import/export.

Команды

cd "Site Repo - ai-mindset-website" npm run dev # Astro, http://localhost:4331 cd studio && npm run dev # Sanity Studio, http://localhost:3343 npm create sanity@latest -- --project ms4mgrt9 --dataset production --template clean npm run export-lab <slug> npm run import-lab content/labs/<slug>.md
секреты не в правила

Переменные окружения

  • SANITY_PROJECT_ID — проект Sanity для CLI.
  • SANITY_DATASET — обычно production.
  • SANITY_API_VERSION — версия API; в repo script по умолчанию 2026-03-29.
  • SANITY_API_TOKEN — только для write/import; временные токены нельзя писать в правила или tracked files.
  • Clean bootstrap команду запускать только в scratch-папке, не внутри активного AIM Website repo.
