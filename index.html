Sanity CMS

Контентный контур AIM Website: Sanity хранит лабы и общие сущности, Astro собирает страницы, product_code связывает тарифы с оплатой, а publish уводит изменения на staging/new.

source Sanity Astro

Контент -> сайт -> оплата

Sanity Studio управляет лабами и общими сущностями. Astro читает Sanity на build-time без CDN, чтобы сразу видеть свежий publish. Тарифы хранят только product_code, а цены приходят рантаймом из платежного контура Дана через `/api/payment/*`.

Studioshared content

Что обновлять в Studio

  • Лабы: документы `lab` с блоками hero, description, pricing, program и reference-блоками.
  • Общие сущности: speakers, cases, reviews, faq, quotes, philosophy, closing phrases.
  • common_content.md: скачивается кнопкой `Экспорт общего контента` и содержит точные reference-ID для агента.
  • External-лабы: редактируются только в Studio; MD-пайплайн работает для internal-лаб.
MD importagent-ready

Лаба через Markdown

  • Шаблон: `create-lab-from-md.md` содержит frontmatter, блоки и чеклист самопроверки.
  • Идентичность: `slug` во frontmatter задаёт URL и документ; имя файла не решает.
  • References: копировать только точные ID из fresh `common_content.md`.
  • Studio upload: `Создать лабу из MD` для новой лабы, `Обновить из MD` внутри существующей.
APIlocal

Как подключаться

cd "Site Repo - ai-mindset-website" npm run dev # Astro, http://localhost:4331 cd studio && npm run dev # Sanity Studio, http://localhost:3343 npm create sanity@latest -- --project ms4mgrt9 --dataset production --template clean # clean scratch Studio npm run export-lab <slug> / npm run import-lab content/labs/<slug>.md
  • Env: `SANITY_PROJECT_ID`, `SANITY_DATASET`, `SANITY_API_VERSION`, `SANITY_API_TOKEN` для write/import.
  • Sanity CLI: clean bootstrap запускать только в scratch-папке; основной Studio уже живёт в `Site Repo - ai-mindset-website/studio`.
  • Freshness: сайт читает Sanity с `useCdn: false`, чтобы rebuild не ловил устаревший CDN.
paymentNocoDB

Как связать оплату

  • product_code: код тарифа из каталога Дана; агент не строит его из slug.
  • Sanity: хранит productCode у каждого pricing plan, но не хранит финальную цену.
  • Runtime: карточки и попап дергают `site-discount-check` через `/api/payment/*`.
  • Fallback: если код не резолвится, карточка показывает `цена уточняется`; для продажи код нужно подтвердить в NocoDB.
deploystaging

Что происходит после publish

  • Webhook: Sanity publish отправляет `repository_dispatch: sanity-content-update`.
  • Build: GitHub Actions запускает Astro build для staging и new.
  • Review: проверять `staging.aimindset.org/labs/<slug>` после rebuild.
  • QA: Vasily и QA-боты работают по documented site/CTA/form/payment rules, не по памяти чата.
agentsvibe coding

Как брать контент для страниц

  • Use: fresh `common_content.md`, Studio export, CLI `export-lab`, Sanity API или tracked docs.
  • Do not use: старый common-content файл, скриншот без проверки или неэкспортированную память чата как финальную истину.
  • Preserve: slugs, dates, reference IDs, product codes, CTA labels and payment links.
  • Handoff: после изменений дать desktop и LAN/mobile preview для проверки.

  • Rule: Alumni discount and promo-code discount do not stack with each other. If both are valid, Alumni wins. Either one can stack with the 5% crypto/USDT payment-method discount.
  • QA: Payment popup tests should cover alumni_only, promo_only, alumni_and_promo, alumni_and_crypto, promo_and_crypto, and alumni_promo_and_crypto.

  • Статус сейчас: `PAUSED`, чтобы ночью ресурс уходил в AIM Night Design Pursuit.
  • Что это: локальный cron в Codex, который может продолжать незавершённую bug-задачу с последнего понятного состояния.
  • Где живёт: `~/.codex/automations/bug/` вне git-репозитория.

  • Что смотреть человеку: если непонятно, почему Codex что-то сам проверял или чинил, сначала открыть `~/.codex/automations/bug/memory.md`.
  • Важно: это исполняющий worker и журнал, а не источник правды для правил сайта; source of truth остаётся в `website-ops/*.json`, `website-ops/*.md`, docs Суриката и QA repo.

  • Case source fidelity: перед каждым новым или существенно обновлённым кейсом, reviewer `Case Source Fidelity Reviewer`.
  • QA dashboards: manual / health, раз в 2 недели по понедельникам 07:00 UTC на четных ISO-неделях, monthly 1-го числа 08:00 UTC.
  • DeviceCloud: manual и preflight 2-го числа 09:00 UTC; реальные сессии ждут provider secrets.
  • Surikat: Cloud Run webhook `aim-surikat-vasily`; local LaunchAgent выключен и используется только как rollback после `deleteWebhook`.
  • Surikat LLM chat: LLM may write only the final short reply after deterministic routing; it must not accept tasks, create Linear issues, or invent QA/dashboard status.
  • Homepage full design: раз в неделю по понедельникам 09:40 Europe/Lisbon; automation `aim-homepage-full-design-generator` создаёт полный light-theme прототип в `homepage-design-lab/runs/full/` и читает Sonya handoff.
  • AIM Night Design Pursuit: каждый день 00:05 Europe/Lisbon; automation `aim-night-design-pursuit` использует ночное окно 00:00–07:00 для разнообразных review-only дизайнов, может закрывать лишние приложения и читает Sonya handoff.
  • Homepage elements: раз в неделю по пятницам 10:20 Europe/Lisbon; automation `aim-homepage-element-design-generator` создаёт один блок/попап/элемент в `homepage-design-lab/runs/elements/` и читает Sonya handoff.
  • Codex bug automation: сейчас `PAUSED`; живёт в `~/.codex/automations/bug/`, при включении продолжает незавершённый bug-thread и пишет след в `memory.md`.
  • Hub: после значимых изменений руками или через sync script; local preview auto-sync подтягивает Product rules, Tool registry и Bot operating rules; desktop/mobile preview обязателен.

Как работает Сурикат Соня

Это не QA-диспетчер и не live deployer, а Telegram taste-memory бот для AIM design labs: она собирает варианты, разговаривает коротко, просит оценки, возвращает сигнал генераторам и передает Василию только багоподобные сигналы.

telegram

Review delivery

  • Где живет: отдельный Telegram-бот `@aim_surikat_sonya_bot`.
  • Хозяева: Анка `@stavenski` и Саша `@alex_named`; numeric id сохраняется после `/start`.
  • Что отправляет: review-only design-lab screenshots с кнопками `0-10` и ссылкой на GitHub Pages/local preview, если она есть.
  • Что можно писать: `/review`, просьбы вроде «пришли дизайны», оценки, медиа-референсы и дизайн-фидбек.
  • Safety gate: scheduled delivery остается approval/catalog-based; прямые owner-команды «все/больше/неоцененное» могут расширять выдачу по inventory без live deploy.
semantic road

Речь и смысл

  • Hard routes: `/start`, `/scan`, `/review`, callbacks и rating replies остаются детерминированными.
  • Обычный текст: идет через Semantic Road: короткое `привет` получает короткий ответ, дизайн-фидбек сохраняется, баг уходит Василию.
  • Cloud path: `SONYA_LLM_PROVIDER=vertex` ведет Semantic Road и scoring через Vertex AI, чтобы использовать Google Cloud billing credits.
  • Связь с Василием: если отзыв похож на баг сайта или сломанную страницу, Соня пишет handoff в очередь Василия вместо прямого Linear-шума.
memory

Что меняется после оценки

  • Registry: каждая оценка пишет `design-feedback-dashboard/data/design-feedback-registry.json`.
  • Generator handoff: `website-ops/design-generator-feedback-handoff.jsonl` получает next-design brief: что сохранить, поменять, запретить или развить.
  • Design Lab: homepage, payment и element generators читают этот handoff перед следующими runs и создают новые review-only страницы, блоки и элементы.
  • Vasily handoff: bug-like feedback по странице/дизайну пишет событие в `Bots/Telegram Bug Intake Bot - Linear/data/sonya-design-handoff.jsonl`.
cloud

Google Cloud режим

  • Entrypoint: `npm run serve` поднимает Cloud Run-compatible server: `GET /health`, `POST /telegram`, protected `POST /review-queue`.
  • Media: `SONYA_GCS_BUCKET` складывает входящие фото/аудио/видео в Cloud Storage; большие файлы для Vertex идут как `gs://...`.
  • Runtime Veo: временный owner-only режим генерирует короткие клипы через Vertex AI с TTL, дневными лимитами и GCS ledger; пока без `ffmpeg` свежие клипы уходят как `sendVideo`, не как настоящий круг.
  • Architecture: `website-ops/sonya-cloud-run-architecture.md`.
  • Deploy notes: `Bots/Telegram Design Taste Bot - Sonya/cloud-run/README.md`.

Как работает Сурикат

Это не отдельный ChatGPT в чате, а тревожный диспетчер QA-контура: он принимает сигналы, решает что является багом, и связывает людей, Linear и автоматические проверки.

telegram

Прием сообщений

  • Где живет: группа сайта и личка `@aim_surikat_bot`.
  • Как обращаться: `Сурикат Василий`, `Сурикат`, `суркат`, `Вася`, `Вась`, `Васек`, `Васёк`, `Васька`.
  • Что ловит: баги, сбои, тормоза, странное поведение, неправильные ссылки, сломанные блоки, проблемы CTA и форм.
  • Как отвечает: если все понятно, ставит `👀`; если мало данных, просит страницу/ссылку, устройство/браузер или скрин.
  • Локальный аудит: все новые входящие update, которые Сурикат реально увидел, пишутся в `data/audit-log.jsonl`.
linear

Маршрут в Linear

  • Проект: `AIM Website`.
  • Статус: `Backlog`.
  • Лейбл: `Баг`.
  • Приоритет: пока расставляется эвристикой из `evaluation-track.md`; дальше можно докрутить на реальных примерах.
  • Если Linear недоступен: отчет сохраняется локально в очередь и будет отправлен после настройки ключа.
edit

Где править Суриката

  • Поведение и речь: `Bots/Telegram Bug Intake Bot - Linear/src/telegram-bug-intake.mjs`.
  • Опции окружения: `.env.example`, `runtime-options.md`.
  • Cloud Run: `Dockerfile`, `cloud-run/README.md`, `cloud-run/env.production.yaml`.
  • Локальный rollback: LaunchAgent `~/Library/LaunchAgents/ai.aim.surikat.bot.plist` запускает `scripts/run-local-watcher.zsh`, но должен быть выключен, пока Telegram webhook смотрит в Cloud Run.
  • Рабочий каталог: `Bots/Telegram Bug Intake Bot - Linear/`.
  • Доступ к сообщениям группы: `group-setup.md`.
  • Priority/Linear: `evaluation-track.md`, `linear-routing.md`.
secrets

Токены и доступы

  • Telegram token: не хранится на сайте и не пишется в репозиторий; если что, токен у Ани.
  • Локально: rollback watcher читает Telegram token из macOS Keychain service `aim_surikat_bot_token`, account `aim_surikat_bot`.
  • Cloud Run: использует Secret Manager `vasily-telegram-bot-token`, `vasily-linear-api-key`, `vasily-telegram-webhook-secret`.
  • Linear: нужен отдельный `LINEAR_API_KEY`; без него Сурикат отвечает в Telegram, но складывает баги в локальную очередь.
  • Команды управления: принимает только от Анки / `@stavenski`; остальных вежливо не слушается.
manager

Связка с QA-ботом сайта

  • Сурикат: диспетчер, голос в Telegram, маршрутизация в Linear, дайджесты людям.
  • AIM Site Agent Evaluation: worker, который запускает black-box проверки, генерирует `reports/latest/report.json` и dashboard.
  • Следующий шаг: команда вроде `Васек, проверь сайт` может запускать GitHub Action, а Сурикат потом будет читать результат и докладывать high/medium findings.
  • Безопасный минимум: пока держим обе системы раздельно: Сурикат принимает человеческие баги, QA-бот регулярно проверяет сайт.
github

GitHub и Actions

  • QA repo: `https://github.com/eppelas/aim-site-agent-evaluation`.
  • QA Actions: `site-qa.yml`, `device-cloud.yml`, `ci.yml`.
  • QA Pages: `https://eppelas.github.io/aim-site-agent-evaluation/`.
  • Surikat runtime: основной runtime теперь Cloud Run service `aim-surikat-vasily`; workflow scaffold `.github/workflows/telegram-bug-intake.yml` остаётся legacy/fallback, не production delivery.

Если run failed

Failure сейчас означает, что найдены high findings или сломался сам прогон. Сначала открываем run status в дашборде.

  • Production: `dashboard.production.html` отделяет боевой сайт от staging.
  • Staging: `dashboard.staging.html` показывает только кандидат на переезд.
  • Evidence: у broken/stale links должна быть source page, anchor text и ближайшая секция.
